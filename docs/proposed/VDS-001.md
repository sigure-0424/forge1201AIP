# VDS-001: Visual Debug System — BOT behaviour visualisation and test tracking

**Status**: Implemented (2026-04-05)
**Branch**: `claude/implement-visual-debug-3xfTo`

## Goal

Make it possible to diagnose why the BOT stops or behaves incorrectly
without a human having to spectate in-game in real time.

Three problems addressed:

1. **Invisible obstacles** — the BOT gets stuck at a block-level step or
   liquid boundary that shows nothing in the log.
2. **No action trace** — `ai_debug_*.json` records snapshots but not
   transitions, so the sequence of events leading to a failure is lost.
3. **No test inventory** — there is no record of which test cases exist,
   which have passed, and which are still failing.

## Implementation Notes

### Step 1 — Action trace logger (`src/debug_trace_logger.js`)
- `logEvent(botId, event, actionName, position, extra)` — appends one JSON
  line per event to `debug_trace.jsonl`.
- `logStatus(botId, payload)` — appends status snapshots.
- Rotates file at 10 MB to `debug_trace.jsonl.bak`.
- Called at action `start`, `complete`, `fail`, and from stuck detection
  with event `stuck`.

### Step 2 — WebSocket debug server (`src/debug_ws_server.js`)
- Port 3001. EADDRINUSE → logs to stderr and continues.
- `broadcast(type, data)` → sends `{type, data, ts}` to all connected
  clients.
- Message types: `path`, `stuck`, `mismatch`, `status`.

### Step 3 — Block registry mismatch detector (`src/debug_mismatch_detector.js`)
- Reads `chunk_dump.json`, calls `bot.blockAt()`, compares `block.name`
  to server-truth name.
- CLI: `node src/debug_mismatch_detector.js [path]` prints dump entries.
- Triggered from `!mismatch` chat command in `bot_actuator.js`.

### Step 4 — WebUI map overlay (in `public/index.html`)
- 600×600 px Canvas, 60×60 block window (30 blocks each direction).
- Layers: block grid, mismatch (red outline), path (yellow line), bot (green
  dot), stuck (red cross, 10 s TTL).
- WebSocket client at `ws://localhost:3001`, auto-reconnects every 3 s.
- `GET /chunk_dump.json` fetched on load and on each `status` event,
  throttled to once per 5 s.

### Step 5 — Test manifest and runner
- `tests/test_manifest.json`: version 1, 11 categories, 19 entries.
- `tests/run_tests.js`: `--category` / `--id` filtering, require-based
  execution, JSON report in `results/`, FAIL→PASS / PASS→FAIL diff.
- `npm run test:runner` script added to `package.json`.

### Step 6 — OBS auto-record on stuck (`src/obs_connector.js`)
- Optional: skipped silently when `OBS_WS_URL` env var is not set.
- Starts recording when `stuckSeconds >= 10`, stops 5 s after bot moves.
- Uses `obs-websocket-js` v5 API.

### Step 7 — Forge client overlay (`aux_mod/…/client/DebugOverlay.java`)
- OkHttp WebSocket client connecting to `ws://localhost:3001`.
- `CopyOnWriteArrayList` for path / mismatch data; `volatile` for stuck /
  bot position.
- `RenderLevelStageEvent` (AFTER_TRANSLUCENT_BLOCKS): yellow path lines,
  red wireframe mismatch cubes, red filled stuck cube (10 s TTL).
- F8 toggle (shared with existing HUD_TOGGLE_KEY binding in `ClientEvents`).
